# 駄菓子事業計画 Loop Engineering 計画書

## 1. 指示書理解サマリ

### プロジェクト目的

- 駄菓子事業計画書の初版を、GitHub Issue駆動かつ証拠駆動で作成できる実行基盤を設計する。
- この段階では、事業計画書本文は作成しない。
- 論点ごとにIssueを分け、情報収集、分析、レビュー、レッドチーム、ブルーチーム、パープルチーム、ジャッジ、ログ記録を通してから最終統合する。

### 事業の前提

- この事業は、単なる駄菓子販売事業ではない。
- 本質は、駄菓子を媒介にして子どもと地域・社会をつなぐ体験事業である。
- 収益最大化を目的にせず、実費回収を基本とした地域貢献・社会体験型の活動として扱う。
- 最初は移動式駄菓子屋として小さく始め、実証実験でニーズ、反応、運営課題、施設側の受け入れやすさを確認する。

### 最終成果物

- レビュー済みIssueの統合用サマリをもとにした「駄菓子事業計画書 初版」。
- 関連Issue、エビデンス、仮説、要検証事項、要調査事項、最終確認チェックリストを追跡できる状態。
- 人間が最終確認できる状態。

### GitHub Issue運用方針

- 親Issueで全体方針、子Issue一覧、Statusルール、証拠駆動ルール、エージェント構成、完了条件を管理する。
- 子Issueは、最終事業計画書へ統合するための素材を作る単位とする。
- 統合前チェックIssue、最終統合Issue、最終レビューIssueを分ける。
- 追加で必要になった論点は、追加Issue候補として記録し、必要に応じて新規Issue化する。

### ステータス管理ルール

- Open / Closedだけで管理しない。
- `Backlog`、`Ready`、`In Progress`、`Blocked`、`Evidence Check`、`Review`、`Changes Requested`、`Integration Ready`、`Integrated`、`Done` を使う。
- `Integration Ready` は統合に使える状態であり、`Done` ではない。
- `Done` は最終統合への反映、反映漏れ確認、Issue完了確認が終わった状態とする。

### 証拠駆動ルール

- すべての主張、判断、提案には根拠を付ける。
- 根拠のない内容は断定しない。
- 決定事項、内部前提、事実、仮説、推論、要検証、要調査を分離する。
- 確信度は高、中、低で示し、次に確認すべきことを記録する。

### エージェント構成

- オーケストレーター
- 要件定義エージェント
- リサーチエージェント
- 分析エージェント
- 作成 / 実装エージェント
- レビューエージェント
- レッドチームエージェント
- ブルーチームエージェント
- パープルチームエージェント
- 統合エージェント
- ジャッジエージェント
- ナレッジ / ログ管理エージェント
- 工数見積もりエージェント

### 各Issueの標準処理フロー

- オーケストレーターがIssueを読み、Statusを進める。
- 要件定義、リサーチ、分析、作成、レビューを実行する。
- 主要Issueではレッドチーム、ブルーチーム、パープルチームを実行する。
- 統合エージェントが統合用サマリを整え、ジャッジエージェントがIntegration Ready可否を判定する。
- ナレッジ / ログ管理エージェントが履歴と判断根拠を残す。

### 各Issueの標準出力テンプレート

- 具体TODO
- 統合用サマリ
- 主張とエビデンス一覧
- 仮説
- 要検証事項
- 要調査事項
- 使用した分析手法
- 使用した専門観点
- レビュー結果
- レッドチーム指摘
- ブルーチーム改善案
- パープルチーム統合判断
- ジャッジ結果
- ナレッジ / ログ
- 次に作るべきIssue候補

### 自動化計画の処理書作成要件

- Issue運用、Issue種別、テンプレート、Status、Label、ステータス遷移、Looper処理、エージェント起動条件、レビュー、チーム処理、統合判定、ログ、エラー処理、人間確認ポイントを定義する。

### エージェント仕様書のMarkdown分割要件

- `agents/` 配下に、1エージェント = 1 Markdownファイルで仕様書を分割する。
- 各仕様書は共通テンプレートに従い、全エージェント共通の禁止事項を含める。
- 本LoopではB方式を採用する。Looper / Orchestratorは各エージェントの役割、起動条件、出力責務を前提知識として保持し、Issueごとに必要なサブエージェントを判断して起動する。
- `agents/*.md` は毎回サブエージェントへ読み込ませるための必須入力ではなく、役割定義の原本、監査基準、Orchestrator実装時の参照仕様として扱う。
- サブエージェントは自分で次のサブエージェントを起動しない。必要な次工程を提案し、起動可否とStatus変更はOrchestratorが判断する。

## 3. 事業の前提

### 決定事項

- 事業は「駄菓子を媒介にして、子どもと地域・社会をつなぐ体験事業」として扱う。
- 最初は移動式駄菓子屋として小さく開始する。
- 最初から本格展開しない。
- 実証実験を通じて、ニーズ、子どもの反応、運営課題、施設側の受け入れやすさを確認する。
- 収益最大化を目的にしない。
- 実費回収を基本とする。

### 要検証事項

- 対象施設が受け入れやすい運営条件。
- 子ども、保護者、施設側、地域団体それぞれの価値認識。
- 運営上の安全、衛生、金銭管理、責任範囲。
- 実費回収に必要な商品構成、価格、販売数。

### 要調査事項

- 出店先施設の利用ルール。
- 食品衛生、アレルギー表示、事故対応、保険などの確認事項。
- 地域イベント、子ども向け活動、類似事例。
- 関係者への説明に必要な資料、許可、同意事項。

## 4. 最終成果物

最終的には、レビュー済みIssueを統合して「駄菓子事業計画書 初版」を作る。ただし、この計画書では本文は作成せず、作成に必要なIssue構造と処理ルールだけを定義する。

最終事業計画書に必要な章立て候補は、最終統合Issueでレビュー済み素材をもとに確定する。現時点では、事業概要、Why、Who、What、How、初回実証実験、商品・価格、運営、収益・費用、施設説明、環境分析、リスク、検証項目、未確定事項、エビデンス、関連Issueなどを統合対象とする。

## 5. GitHub Issue階層

```text
親Issue: 駄菓子事業計画 初版作成
  ↓
子Issue: 論点ごとの調査・分析・素材化
  ↓
統合前チェックIssue
  ↓
最終統合Issue: 事業計画書 初版作成
  ↓
最終レビューIssue
  ↓
追加Issue候補
```

### 親Issue

- プロジェクト目的、事業前提、子Issue一覧、Statusルール、証拠駆動ルール、エージェント構成、完了条件を管理する。
- 自動化計画の処理書と各エージェント仕様書へのリンクを持つ。
- Orchestratorが保持するエージェント役割マップの基準を管理する。

### 子Issue

- 論点ごとに統合用素材を作る。
- 事業計画書本文ではなく、最終統合に使うサマリ、エビデンス、仮説、要検証、要調査を作る。
- 各子Issueでレビューエージェントを必ず起動する。
- 主要Issueではレッドチーム、ブルーチーム、パープルチームを起動する。

### 統合前チェックIssue

- 各子IssueがIntegration Ready条件を満たしているか確認する。
- 未達Issueは `Changes Requested` または `Blocked` に戻す。

### 最終統合Issue

- Integration ReadyのIssueだけを使って、最終事業計画書の初版素材へ統合する。
- 統合元Issueを追跡できるようにする。
- 新しい主張を勝手に追加しない。

### 最終レビューIssue

- 完成基準、根拠、抜け漏れ、仮説と事実の分離、人間確認ポイントを確認する。

## 6. GitHub Issueステータス管理

| Status | 意味 | 次に進む条件 |
|---|---|---|
| Backlog | 作成済みだが、まだ着手対象ではない | 着手優先度と必要入力が確認された |
| Ready | 着手可能 | オーケストレーターが着手対象に選んだ |
| In Progress | Looperまたはサブエージェントが対応中 | 統合用素材のドラフトができた |
| Blocked | 情報不足、判断待ち、外部確認待ち | 不足情報または判断が得られた |
| Evidence Check | 根拠、証拠、分類の確認待ち | 主張分類とエビデンス表が整った |
| Review | レビュー待ち | レビューエージェントが確認した |
| Changes Requested | レビューで修正が必要 | 修正後にIn Progressへ戻す |
| Integration Ready | 統合用サマリが完成し、最終統合に使える | 最終統合Issueで反映された |
| Integrated | 最終事業計画書に反映済み | 反映漏れ確認が完了した |
| Done | Issueとして完了 | 反映、確認、ログ記録が完了した |

### 基本遷移

```text
Backlog
  ↓
Ready
  ↓
In Progress
  ↓
Evidence Check
  ↓
Review
  ↓
Integration Ready
  ↓
Integrated
  ↓
Done
```

### 修正時の遷移

```text
Review
  ↓
Changes Requested
  ↓
In Progress
  ↓
Evidence Check
  ↓
Review
```

### Blocked時の記録

```md
## Blocked理由

### 何が不足しているか
-

### 誰の判断が必要か
-

### どの情報があれば再開できるか
-

### 追加Issueが必要か
-
```

### Integration Ready条件

- Issueの目的に対応した成果物がある。
- 統合用サマリがある。
- 主張とエビデンス一覧がある。
- 仮説、要検証事項、要調査事項が分離されている。
- レビューエージェントの確認が完了している。
- 主要Issueではレッドチームの検証が完了している。
- 主要Issueではブルーチームの改善案が反映または整理されている。
- 主要Issueではパープルチームの統合判断がある。
- ジャッジエージェントがIntegration Ready可と判定している。
- 最終事業計画書に反映すべき内容が明確である。

### Done条件

- `Integration Ready` になっている。
- 最終統合Issueで事業計画書素材へ反映されている。
- Statusが `Integrated` になっている。
- 反映漏れ確認が完了している。
- ナレッジ / ログ管理エージェントが履歴を記録している。
- Issueとして完了できるとジャッジエージェントが判定している。

## 7. 証拠駆動ルール

### 主張分類

| 分類 | 意味 | 扱い |
|---|---|---|
| 決定事項 | ユーザーまたは既存資料で明確に決まっていること | 原則として高確信度 |
| 内部前提 | 既存資料、壁打ちメモ、既存タスクに基づくこと | 根拠元を明示する |
| 事実 | 公開情報、公式情報、調査資料などで確認できること | 出典または確認方法を明示する |
| 仮説 | 未検証だが計画上置いておく考え | 断定しない |
| 推論 | 複数の前提からAIが導いた判断 | 前提と推論過程を明示する |
| 要検証 | 実証実験、ヒアリング、現場確認で検証すべきこと | 実証実験設計へ渡す |
| 要調査 | 法規制、施設ルール、衛生、地域事情など追加調査が必要なこと | 追加Issue化を検討する |

### 確信度

| 確信度 | 意味 |
|---|---|
| 高 | 決定事項、内部前提、明確な事実に基づく |
| 中 | 前提情報から合理的に推論できる |
| 低 | 仮説であり、検証または調査が必要 |

## 9. エージェント構成

本Loopでは、Looper / Orchestratorが以下のエージェント構成を前提知識として理解しているものとして扱う。各エージェント仕様書は、Orchestratorの判断基準を外部化した仕様書であり、毎回のサブエージェント起動時に必ず読み込ませるものではない。

| No | エージェント | 役割 | 起動タイミング |
|---|---|---|---|
| 1 | オーケストレーター | 全体進行、Issue分解、Status管理、各エージェント起動 | 全Issue |
| 2 | 要件定義エージェント | 目的、前提、制約、成果物、スコープを整理 | 子Issue開始時 |
| 3 | リサーチエージェント | 内部前提、外部情報、参考事例、確認事項を収集 | 情報が必要なIssue |
| 4 | 分析エージェント | Issue目的に応じて分析手法を選定し分析 | 子Issueの情報整理後 |
| 5 | 作成 / 実装エージェント | Issue成果物と統合用素材を作成 | 分析後 |
| 6 | レビューエージェント | 要件、抜け漏れ、根拠、分類を確認 | 各Issueで必須 |
| 7 | レッドチームエージェント | 失敗要因、反論、リスク、前提破綻を検証 | 主要Issue |
| 8 | ブルーチームエージェント | 指摘への改善策、代替案、要検証化を整理 | レッドチーム後 |
| 9 | パープルチームエージェント | 指摘と改善案を統合し修正方針を決める | レッド / ブルー後 |
| 10 | 統合エージェント | 統合用サマリを整え、追跡可能にする | Integration Ready判定前、最終統合時 |
| 11 | ジャッジエージェント | Integration Ready、Done可否を判定 | レビュー後、統合後 |
| 12 | ナレッジ / ログ管理エージェント | 決定事項、変更履歴、未解決論点、反映元を記録 | 各Issue終盤 |
| 13 | 工数見積もりエージェント | WBS、3点見積もり、AI短縮工数、人間確認工数を整理 | 必要に応じて |

## 10. 各Issueの処理フロー

1. オーケストレーターがIssueを読む。
2. Orchestratorが保持するエージェント役割マップに基づき、必要なサブエージェントと順序を判断する。
3. Statusを `Ready` から `In Progress` へ変更する。
4. 要件定義エージェントが目的、前提、制約、成果物、スコープを整理する。
5. リサーチエージェントが必要情報を収集する。
6. 分析エージェントがIssue目的に応じて分析手法を選定する。
7. 分析エージェントが選定理由、不採用理由、分析結果、仮説、要検証、要調査を出す。
8. 作成 / 実装エージェントが統合用素材を作成する。
9. Statusを `Evidence Check` へ変更し、主張分類とエビデンス表を確認する。
10. Statusを `Review` へ変更し、レビューエージェントを起動する。
11. 主要Issueではレッドチームが失敗要因、反論、リスク、前提破綻を検証する。
12. ブルーチームがレッドチーム指摘への改善策、代替案、要検証化、追加Issue案を作る。
13. パープルチームが採用指摘、反映改善案、追加Issue化、修正方針を決める。
14. 統合エージェントが統合用サマリを整える。
15. ジャッジエージェントが `Integration Ready` 可否を判定する。
16. ナレッジ / ログ管理エージェントが履歴を記録する。
17. 必要に応じて工数見積もりエージェントが工数を整理する。
18. 条件を満たせばStatusを `Integration Ready` へ変更する。
19. 要修正ならStatusを `Changes Requested` へ変更する。
20. 情報不足ならStatusを `Blocked` へ変更する。

## 11. 最終統合ルール

- 統合対象は `Integration Ready` のIssueに限定する。
- レビュー未完了、エビデンス未確認、ジャッジ未完了のIssueは統合しない。
- 統合エージェントは、新しい主張を追加しない。
- 事業計画書へ反映する内容、反映しない内容、追加仮説、要検証、要調査を分ける。
- 統合元Issueの番号、タイトル、Status、反映箇所を記録する。
- 法規制、衛生、責任範囲、人間判断が必要な事項は、確定事項として扱わない。
- 最終レビューIssueで、根拠、抜け漏れ、仮説と事実の分離、要検証・要調査、人間確認ポイントを確認する。
