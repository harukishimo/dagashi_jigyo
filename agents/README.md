# Agent Specifications

## 目的

このディレクトリは、駄菓子事業計画 Loop Engineeringで使用する各エージェントの仕様書を管理する。

各仕様書は、1エージェント = 1 Markdownファイルで分割する。事業計画書本文を作成するためのファイルではなく、Issue駆動・証拠駆動で事業計画書を作成するための実行基盤である。

## サブエージェント起動方式

本LoopではB方式を採用する。

- Looper / Orchestratorは、各エージェントの役割、起動条件、入力、出力、禁止事項を前提知識として保持しているものとして扱う。
- `agents/*.md` は、毎回サブエージェントへ読み込ませるための必須入力ではなく、役割定義の原本、監査基準、実装時の参照仕様である。
- サブエージェントは、自分で別のサブエージェントを起動しない。
- サブエージェントは、次に必要な工程やエージェントを提案できる。
- 実際の起動可否、順序変更、Status変更はOrchestratorが判断する。

## 基本処理フロー

1. オーケストレーターがIssueを読み、Statusと必要エージェントを確認する。
2. Orchestratorが保持する役割マップに基づき、必要なサブエージェントと順序を判断する。
3. 要件定義エージェントが目的、前提、制約、成果物、スコープを整理する。
4. リサーチエージェントが必要情報を収集し、未確認情報を要調査に分ける。
5. 分析エージェントがIssue目的に応じて分析手法を選定する。
6. 作成 / 実装エージェントが統合用素材を作成する。
7. レビューエージェントが各Issueで必ず確認する。
8. 主要Issueではレッドチーム、ブルーチーム、パープルチームを実行する。
9. 統合エージェントが統合用サマリを整える。
10. ジャッジエージェントがIntegration Ready可否を判定する。
11. ナレッジ / ログ管理エージェントが履歴を記録する。

## GitHub Issueステータスとの関係

- `Backlog`: Issue作成済み、未着手。
- `Ready`: 着手可能。
- `In Progress`: エージェント処理中。
- `Blocked`: 情報不足、人間判断待ち、外部確認待ち。
- `Evidence Check`: 主張分類と根拠確認中。
- `Review`: レビュー待ち。
- `Changes Requested`: 修正待ち。
- `Integration Ready`: 統合可能。
- `Integrated`: 最終事業計画書素材に反映済み。
- `Done`: 反映確認とログ記録まで完了。

## 証拠駆動ルール

- 決定事項、内部前提、事実、仮説、推論、要検証、要調査を分離する。
- 各Issueでエビデンス表を作成する。
- 根拠のない断定を禁止する。
- 未確認情報は要検証または要調査として扱う。

## レッド / ブルー / パープルチーム運用

- レッドチームは主要Issueで失敗要因、反論、リスク、前提破綻を探す。
- ブルーチームはレッドチーム指摘への改善策、代替案、要検証化を行う。
- パープルチームは採用指摘、反映改善案、追加Issue化、保留内容を決める。

## 最終統合時の役割分担

- 統合エージェントは `Integration Ready` のIssueだけを扱う。
- ジャッジエージェントは統合可否、Done可否を判定する。
- ナレッジ / ログ管理エージェントは統合元Issueと反映履歴を記録する。
- レビューエージェントは最終レビューで根拠、抜け漏れ、仮説と事実の分離を確認する。
